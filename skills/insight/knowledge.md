# 知识图谱 & 树搜索

> 图谱不独立存在——从索引派生，嵌入搜索管线，驱动质量检查。
> 树搜索不替代摘要——摘要管「读不读」，树管「读哪里」。

## 触发词

```
「文档关联」「知识图谱」「文档之间怎么关联」
「找相关文档」「分析文档关系」「查看图谱」
```

## 一、共现图

### 1.1 数据来源：索引即图

不从零构建。直接从已有索引文档的 entries 提取，**零 LLM 成本**。

```
索引文档：SpringBoot
  entries: [doc1(w=10), doc3(w=9), doc7(w=6)]

索引文档：自动配置
  entries: [doc1(w=9), doc5(w=7)]

→ 共现：SpringBoot 和 自动配置 共享 doc1 → 有边
```

### 1.2 图模型

| 元素 | 来源 | 含义 |
|------|------|------|
| 节点 | 索引文档标题（关键词） | 一个知识概念 |
| 边 | 两个关键词共享 ≥1 篇源文档 | 概念间存在关联 |
| 边权重 | 共享文档数 × 双方权重调和平均 | 关联强度 |
| 节点大小 | 该关键词的 entries 总数 | 概念覆盖广度 |

### 1.3 构建

```
1. yuque_list_docs → 子索引库全部索引文档（标题 + id）
2. 5 并发 yuque_get_doc → 读每个索引文档 body
3. parseIndexDoc → 提取 entries 数组
4. 纯计算：
   - 建倒排：doc_id → [keyword1, keyword2, ...]
   - 对每对共现关键词：累加共享文档数 + 权重调和平均
   - 输出：节点列表 + 边列表
5. 可选：跑 Louvain 社区检测 → 自动发现知识聚类
```

> 不上 LLM。第 4-5 步纯数据处理，毫秒级。

### 1.4 存储

图谱作为子索引库中的一篇索引文档，标题固定 `_graph`，body 为 JSON。

```json
// _graph 索引文档 body
{
  "built_at": "2026-06-02T20:00:00Z",
  "nodes": 156,
  "edges": 423,
  "communities": [
    {"id": 0, "label": "Spring生态", "keywords": ["SpringBoot","自动配置","JPA"], "cohesion": 0.72},
    {"id": 1, "label": "容器化", "keywords": ["Docker","K8s"], "cohesion": 0.85}
  ]
}
```

> 跟其他关键词索引文档一样，通过 `yuque_kb_search(["_graph"])` 定位，`yuque_get_doc` 读取。

### 1.5 查看图谱

用户说「查看图谱」→ `yuque_kb_search(["_graph"])` → 找到就读 → 没有就现场从索引 entries 算一份 → 输出报告。

报告内容：社区概览表 + 孤立关键词 + 可选 Mermaid 图。不存成独立文档，直接回复。

## 二、搜索增强

### 2.1 图谱扩展

在搜索管线中插入一步——命中关键词后，用图找到关联关键词，补搜一轮。

```
原流程：token 搜索引 → 展开 entries → 重排 → 读原文 → 生成答案

增强后：token 搜索引 → 展开 entries
           ↓
        yuque_kb_search(["_graph"]) 找图谱文档
          → 命中关键词的 1 跳邻居（权重前 5）
          → 补搜邻居关键词的索引文档
          → 展开新 entries → 去重合并
           ↓
        重排 → 读原文 → 生成答案
```

### 2.2 为什么是 1 跳

| 跳数 | 召回 | 噪音 | 延迟 |
|------|------|------|------|
| 1 跳 | +30-50% | 低（权重过滤） | +1 轮搜索 |
| 2 跳 | +5-10% | 高（关联稀释） | +2-3 轮搜索 |

1 跳够了。2 跳性价比太低。

### 2.3 何时触发

不是每次搜索都开图扩展。满足以下条件之一才触发：

| 条件 | 逻辑 |
|------|------|
| 直接命中 < 3 篇 | 候选太少，图扩展补量 |
| 用户问题涉及「关系」「对比」「关联」 | 语义上需要关联知识 |
| 子索引库 graph 字段存在且新鲜（built_at > 30 天前） | 图数据可用 |

### 2.4 search.md 集成

在 search.md 步骤 ② 和 ③ 之间插入 `②.5 图谱扩展（条件触发）`。详见 search.md 修改。

## 三、索引质量

不用 LLM 判断，用图结构硬指标。

### 3.1 孤立关键词

```
定义：该关键词的 degree = 0（不与任何其他关键词共现）
含义：这个关键词只有一篇文档 → 可能索引粒度太细或文档太偏
行动：提示合并到上级关键词，或保留（如果是独立领域入口）
```

### 3.2 弱关联关键词

```
定义：degree > 0 但所有边权重 < 2（共享文档数太少）
含义：概念有微弱联系但不够紧密
行动：检查是否该合并、拆分、或扩充文档
```

### 3.3 关键词覆盖盲区

```
已知：子索引库 description.source_books 里的所有源文档
实际：所有索引文档 entries 里出现过的 doc_id 集合
盲区 = 已知 - 实际 = 有文档但没被任何关键词索引覆盖
```

### 3.4 稀疏社区

```
定义：Louvain 社区 cohesion < 0.3
含义：算法把它们归为一类，但实际上内部联系很弱
行动：可能是索引关键词选择偏差，提示人工复查
```

### 3.5 质量报告

每次索引构建完成后自动跑一次质量检查，产出：

```
📊 索引质量报告

节点：156 关键词
边：423 条关联
社区：8 个知识聚类

⚠️ 孤立关键词（3）：RxJava、WebFlux、gRPC
   建议：检查是否需要合并或补充文档

⚠️ 文档覆盖盲区（2 篇）：did=123、did=456
   原因：索引构建时这些文档未被任何关键词匹配

✅ 稀疏社区：无（最低 cohesion 0.52，健康）
```

## 四、树搜索

### 4.1 场景

当一篇源文档 > 5000 字时，读全文浪费 token 且稀释答案精度。树搜索让 LLM 只读相关章节。

### 4.2 数据模型

在索引文档 entry 中增加可选字段 `tree`：

```json
{
  "title": "Canvas 绘图实战",
  "keywords": ["Canvas", "Canvas2D", "层次关系图"],
  "search_surface": "Canvas怎么绘图，Canvas画层次关系图教程",
  "summary": "本文介绍Canvas指令式绘图基础...",
  "doc_id": 584,
  "namespace": "yehuoshun/huwsx0",
  "doc_title": "Canvas 绘图实战",
  "slug": "canvas-draw",
  "url": "https://www.yuque.com/yehuoshun/huwsx0/canvas-draw",
  "weight": 9,
  "tree": {
    "sections": [
      {"id": "s1", "title": "Canvas 基础概念", "summary": "坐标系、绘图上下文、基本API"},
      {"id": "s2", "title": "绘制层次关系图", "summary": "实战：用Canvas绘制树形层次结构"},
      {"id": "s3", "title": "性能优化策略", "summary": "离屏Canvas、分层渲染、requestAnimationFrame"}
    ]
  }
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `tree` | ❌ | 章节树，文档 ≤ 5000 字可省略 |
| `sections[].id` | ✅ | 章节标识，短字符串 |
| `sections[].title` | ✅ | 章节标题 |
| `sections[].summary` | ✅ | 一句话概括章节内容（≤50 字） |

### 4.3 建树

索引构建时，逐文档评分那步顺带做。一次 LLM 调用同时输出关键词 + 章节树：

```json
// LLM 输出格式（在逐文档评分的基础上追加 tree）
{
  "keywords": [{"keyword": "Canvas", "w": 10}, {"keyword": "Canvas2D", "w": 8}],
  "tree": {
    "sections": [
      {"id": "s1", "title": "Canvas 基础概念", "summary": "坐标系、绘图上下文、基本API"},
      ...
    ]
  }
}
```

> 文档 ≤ 5000 字可不输出 tree，省 token。

### 4.4 搜索流程

替换原步骤 ⑤（读全文），改为先读树再选章节：

```
原步骤 ⑤：
  5 并发 yuque_get_doc 读源文档全文 → LLM 生成答案

树搜索版步骤 ⑤：
  ⑤-a. 对每篇候选：检查 entry 有无 tree 字段
        ├─ 有 tree → 进入 ⑤-b
        └─ 无 tree → 降级读全文

  ⑤-b. LLM 读 tree.sections（只有标题+摘要，几十 token）
        → 判断哪些 section 与用户问题相关
        → 输出相关 section id 列表

  ⑤-c. 只读相关 section 的原文
        → 拼接后送 LLM 生成答案 + 引用
```

### 4.5 与摘要的关系

| | 摘要 (esum) | 树 (tree) |
|------|------|------|
| 在哪 | 索引文档 entry 里 | 同上，可选字段 |
| 谁用 | LLM 重排阶段 | LLM 读原文阶段 |
| 粒度 | 整篇文档 | 章节级 |
| 作用 | 判断「这篇值不值得读」 | 判断「读这篇的哪里」 |
| 互斥？ | ❌ 互补 | ❌ 互补 |

### 4.6 Token 节省估算

| 文档字数 | 读全文 token | 读树 + 2 个相关章节 token | 节省 |
|------|------|------|------|
| 5000 字 | ~3500 | ~200 + 1500 | ~50% |
| 10000 字 | ~7000 | ~200 + 2000 | ~70% |
| 20000 字 | ~14000 | ~200 + 3000 | ~78% |

文档越长越划算。

## 五、索引文档 entry 字段扩展

在现有 entry 字段基础上追加：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `tree` | `{sections: [{id, title, summary}]}` | ❌ | 章节树，文档 ≤ 5000 字可省略 |

> 不影响现有索引文档格式，`tree` 为可选字段。旧索引不需要重建。
