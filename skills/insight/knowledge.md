# 文档关联图谱（knowledge-graph）

自动发现文档间隐藏关联，构建知识库关联图谱，支持增量更新。

## 触发词

```
「文档关联」「知识图谱」「文档之间怎么关联」
「这篇跟哪些文档相关」「找相关文档」「交叉引用」
「给知识库建关联图」「分析文档关系」
```

---

## 三种模式

| 模式 | 输入 | 输出 | 持久化 |
|------|------|------|--------|
| 🔗 单篇关联 | 1 篇文档 | 相关文档列表 + 关联理由 | 写入图谱文档（新增/更新行） |
| 🕸️ 全库图谱 | 知识库 | 关联关系图 + 聚类分组 | 写入/重建图谱文档 |
| ✏️ 自动补引用 | 1 篇文档 | 文中插入交叉引用链接 | 更新原文（⚠️ 需确认） |

---

## 持久化策略（方案 C）

```
图谱文档：每个知识库对应一篇 _知识图谱_{库名}，存所有关联关系
原文引用：仅在用户明确要求「补引用」「插入链接」时才改原文

文件位置：
  图谱文档：创建在目标知识库中，标题 _知识图谱_{库名}
  增量更新：yuque_update_doc 追加新行，不覆盖已有
  全量重建：全库图谱模式 → 覆盖重建图谱文档
```

> ⚠️ 图谱文档可能被用户误操作。运行前先 `yuque_list_docs` 检查是否已存在，存在则增量合并。

---

## 图谱文档格式

```markdown
# 🕸️ 知识图谱：《{知识库名}》
> 自动生成 | 最后更新：{时间} | 覆盖 {N} 篇文档

## 📊 主题聚类
| 主题 | 文档数 | 文档 |
|------|--------|------|
| 鉴权体系 | 5 | JWT实战、JWT安全原理、OAuth2、SSO、权限模型 |
| 容器化 | 3 | Docker入门、Docker Compose、K8s基础概念 |

## 🔗 关联关系
| 源文档 | 类型 | 目标文档 | 强度 | 关联理由 |
|--------|------|---------|------|---------|
| JWT实战 | 前置知识 | JWT安全原理 | 强 | 实战用到了安全原理中的 token 设计 |
| Docker入门 | 互补 | Docker Compose | 强 | 合起来覆盖完整容器化部署流程 |
| Redis基础 | 概念引用 | 分布式锁 | 中 | 分布式锁实现依赖 Redis 命令 |

## 📎 建议交叉引用（待确认）
| 源文档 | 建议引用 | 位置 | 原因 |
|--------|---------|------|------|
| JWT实战 | JWT安全原理 | 第3段-签发流程 | 文中提到 token 安全性但未引用详细说明 |

## 🏷️ 孤立文档（无关联）
| 文档 | 最后更新 | 建议 |
|------|---------|------|
| 旧笔记 | 2023-01-01 | 可能已过时，建议归档 |
```

## 模式一：🔗 单篇关联

### 流程

```
1. yuque_get_doc → 读目标文档（提取：核心概念、关键术语、主题）
2. yuque_list_docs → 获取同知识库候选文档（最多 30 篇，排除目标本身）
3. 并发 yuque_get_doc（并发 5）→ 读候选 body
4. LLM 逐对分析 → 从 4 维度判定关联

4 维判定标准：
  主题重叠：讨论同一技术/概念/领域
  依赖关系：A 是 B 的前置知识 / B 引用了 A 的结论
  概念引用：正文中出现另一篇文档的核心术语
  互补关系：两篇合在一起覆盖完整知识面（如上下篇）

5. 去重排序 → 按关联强度降序
6. 写入图谱文档（检查存在 → 有则更新关联关系表相关行，无则新建）
```

### 输出

```
🔗 《{title}》关联分析

📊 共发现 {N} 篇关联文档：

🟢 强关联（{n}）：
  《A》— 前置知识：本文用到了 A 中的核心概念
  《B》— 互补：本文 + B 覆盖完整的部署方案

🟡 中关联（{n}）：
  《C》— 概念引用：文中提到但未展开

⚪ 弱关联（{n}）：
  《D》— 同主题，不同角度

📎 已更新图谱文档：https://www.yuque.com/{namespace}/...
```

## 模式二：🕸️ 全库图谱

### 流程

```
1. yuque_list_docs → 全部文档元数据（title/id/updated_at/word_count）
   上限 100 篇，超出提示分批

2. LLM 基于标题 + word_count 初筛聚类：
   → 不读全部 body，用元数据聚类出候选组
   → 输出主题分组表

3. 组内精细分析：
   → 组内文档数量：> 5 篇则只读前 5 篇做精细关联
   → 并发 yuque_get_doc → LLM 判定组内关联

4. 识别孤立文档：无关联的文档单独列出

5. 生成图谱文档 → yuque_create_doc 或 yuque_update_doc
```

### 输出

```
🕸️ 知识库关联图谱：《{库名}》（{N} 篇）

📊 主题聚类
  鉴权体系（5篇）：JWT实战、JWT安全原理...
  容器化（3篇）：Docker入门、Docker Compose...

🔗 关联关系（{M} 条）
  [强] JWT实战 ← 前置 — JWT安全原理
  [强] Docker入门 ← 互补 — Docker Compose
  [中] Redis基础 ← 引用 — 分布式锁
  [弱] SpringBoot基础 ← 同主题 — SpringCloud入门

🏷️ 孤立文档（{K} 篇）
  旧笔记（2023-01-01）— 建议归档
  TODO草稿（2024-06-01）— 待完成

📎 建议交叉引用（{P} 条）
  1. JWT实战 → 应引用 JWT安全原理 — 位置：第3段签发流程

✅ 图谱已保存：https://www.yuque.com/{namespace}/_知识图谱_{库名}
```

## 模式三：✏️ 自动补引用

### 流程

```
1. yuque_get_doc → 读目标文档 body
2. yuque_list_docs → 获取候选文档列表
3. LLM 分析 → 找出正文中哪些位置应该加交叉引用
4. 输出预览（原文 + 🔗 标注新引用位置）
5. ⚠️ 用户确认后 → yuque_update_doc 写入
```

### 输出预览格式

```
原文第 3 段：
  JWT token 的安全性需要特别注意...

🔗 建议插入引用：《JWT 安全原理》— 详细介绍了 token 安全设计

原文第 7 段：
  部署时使用 Docker Compose 编排多个服务...

🔗 建议插入引用：《Docker Compose 多服务编排》— 完整的 compose.yml 示例

---
确认后将在这 {N} 处插入引用链接。确认？
```

## 增量更新策略

```
图谱文档已存在时：
  1. yuque_get_doc 读图谱文档 body
  2. 解析已有关联关系表（Markdown 表格）
  3. 新分析结果与已有数据合并：
     - 相同关联 → 跳过
     - 新关联 → 追加行
     - 冲突关联（同一对文档，不同结论）→ 用最新结果覆盖
  4. yuque_update_doc 写回

图谱文档不存在时：
  → yuque_create_doc 新建
```

## 上限保护

| 模式 | 候选上限 | 说明 |
|------|---------|------|
| 单篇关联 | 30 篇 | 读 30 篇 body 做逐对分析 |
| 全库图谱 | 100 篇 | 元数据聚类 + 组内最多 5 篇细读 |
| 自动补引用 | 20 篇 | 候选文档范围 |

---

## 错误处理

| 场景 | 处理 |
|------|------|
| 文档未找到 | 「未找到文档，检查名称后重试」 |
| 知识库为空 | 「知识库是空的，没有文档可做关联分析」 |
| 候选文档全为空 | 「未发现有效关联，可能文档间主题差异过大」 |
| 图谱文档更新失败 | 降级为直接返回分析结果 |
| 读取失败(429) | 等 Retry-After，最多重试 3 次 |
